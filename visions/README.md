# ビジョン

このプロジェクトは「GitHub のような *Git リモート*」を自前で提供しつつ、特に日本語で書かれがちな **要件定義書・仕様書** と **開発ルール** を中心に、開発の意思決定と品質状況を“見える化”できる Web アプリをつくることを目指す。

## 解きたい課題

- 仕様（要件定義書）やルール（セキュリティ、コーディング規約、テスト方針など）が散逸し、参照されず形骸化しやすい
- 仕様・ルールと、実装・テスト・品質指標の関係が追いづらい
- 日本語ドキュメントの量が増えるほど「探せない・揃わない・更新されない」状態になりやすい

## 目指す姿

- ふだん通り `git clone / commit / push` するだけで、サーバ側にリポジトリが保存され、Web からも閲覧できる
- リポジトリ内の `specs/` と `rules/` を **第一級の成果物**として扱い、読みやすい UI（目次、検索、差分）で表示できる
- `rules/` は分類（security/coding/testing…）が分かる形で管理され、**ルール種類の割合**を「言語割合」のように可視化できる
- テストカバレッジなどの品質指標が、プロジェクトの状態として分かりやすく表示される

## 重要な方針（思想）

- **Docs-as-code**: 仕様・ルールはリポジトリに置き、変更は履歴に残す
- **規約の強制**: `specs/` や `rules/` などの必須構造は push 時に検証して担保する（運用ではなく仕組みで守る）
- **低摩擦**: 特別なクライアントを要求せず、標準の Git 体験を崩さない
- **段階導入**: まずは「Git リモート + 可視化」を成立させ、将来の拡張（認証強化、非公開 repo、AI 支援）につなげる

## 将来の拡張（構想）

- 仕様書駆動開発を支えるためのトレーサビリティ（仕様項目 ↔ 変更 ↔ テスト）
- ユーザー管理・トークン発行（PAT）・権限制御
- AI を組み込む場合でも、`specs/` と `rules/` が整備されていることで「参照できる一次情報」を増やし、開発の質を上げる土台にする
